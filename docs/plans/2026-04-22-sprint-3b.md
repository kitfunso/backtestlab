# Sprint 3b — Supabase auth + cross-device sync + Resend rebalance alerts

**Goal:** Turn saved portfolios into durable, cross-device objects. Add Supabase email-magic-link auth, replace localStorage with a `user_portfolios` table, and ship per-portfolio rebalance alerts via Resend email triggered by a daily GitHub Actions cron.

**Prerequisites:**
- Sprint 3a shipped at v0.3.1 (fundamentals + DCA + disclaimer audit)
- localStorage-backed saved portfolios working (v1 schema: `bl.portfolios.v1`)
- $0/month infra target still binds

**Decisions locked:**
- **Auth provider:** Supabase email magic link. No passwords. No social logins in v1.
- **Alert scheduler:** GitHub Actions daily cron (matches ingest-mcx + fetch-news + build-factors pattern). Not a Cloudflare Worker, not a Supabase edge function — one scheduling primitive across the repo.
- **Email provider:** Resend free tier. Sender domain TBD (`alerts@backtest-lab.com` pending DNS setup; falls back to `onboarding@resend.dev` if DNS not ready).
- **Sync strategy:** last-write-wins using `updated_at`. Conflict resolution is out of scope (users rarely edit a portfolio on two devices simultaneously).
- **Migration:** on first sign-in, localStorage portfolios upload to DB and localStorage becomes a read-through cache. No delete on migrate.

**Out of scope (explicitly):**
- Password auth, OAuth, 2FA
- Push notifications (mobile PWA is Sprint 4+)
- Real-time alert webhooks (only daily email in v1)
- Paid tiers / Stripe integration
- Social features (share portfolio, friend list)

**Estimated scope:** 14 steps, ~2 working sessions with CC.

## Part A — Supabase auth

### Step 1: Supabase project + schema

**Files:**
- `supabase/schema.sql` (new)
- `supabase/README.md` (new — one-page setup guide)

**What:**
- Manual: create Supabase project at `backtestlab.supabase.co` (user action, ~5 min)
- SQL schema:
  ```sql
  create table user_portfolios (
    id uuid primary key default gen_random_uuid(),
    user_id uuid not null references auth.users(id) on delete cascade,
    name text not null,
    spec_json jsonb not null,
    created_at timestamptz not null default now(),
    updated_at timestamptz not null default now(),
    unique (user_id, name)
  );
  create table portfolio_alerts (
    id uuid primary key default gen_random_uuid(),
    portfolio_id uuid not null references user_portfolios(id) on delete cascade,
    user_id uuid not null references auth.users(id) on delete cascade,
    kind text not null check (kind in ('drawdown','deviation')),
    threshold_pct numeric not null,
    last_fired_at timestamptz,
    created_at timestamptz not null default now()
  );
  alter table user_portfolios enable row level security;
  alter table portfolio_alerts enable row level security;
  create policy own_portfolios on user_portfolios
    for all using (auth.uid() = user_id) with check (auth.uid() = user_id);
  create policy own_alerts on portfolio_alerts
    for all using (auth.uid() = user_id) with check (auth.uid() = user_id);
  ```
- README documents: project URL, anon key, service-role key env var names, magic-link redirect URL (must include `https://backtestlab.pages.dev/*` in Supabase auth settings).

**Verify:** psql to Supabase, `select * from user_portfolios;` returns 0 rows. Auth settings show magic-link enabled.

**Commit:** `infra(supabase): schema + RLS for user_portfolios and portfolio_alerts`

### Step 9: Python alert evaluator

**Files:**
- `scripts/evaluate_alerts.py` (new)
- `scripts/requirements.txt` (edit — add `supabase==2.3.4` and `resend==0.8.0` if the latter exists; otherwise use `requests` against Resend REST API)

**What:** Script that:
1. Uses service-role key to read all `portfolio_alerts` where `last_fired_at IS NULL OR last_fired_at < now() - interval '24 hours'`
2. For each alert's `portfolio_id`, loads the `spec_json` and computes the current portfolio value from `public/india/prices/*.json` + `public/india/prices/mcx/*.json`
3. Reconstructs equity curve from portfolio inception (`created_at`) → today; computes drawdown-from-peak and per-asset deviation-from-target
4. If threshold crossed, marks alert for firing; records the trigger reason + numbers
5. Returns a list of (user_email, alert_summary) tuples for the email step

**Verify:**
```bash
python scripts/evaluate_alerts.py --dry-run
```
Expect: logs how many alerts evaluated, how many triggered, no emails sent, no DB writes.

**Commit:** `feat(alerts): python evaluator for drawdown + deviation thresholds`

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Resend free tier caps at 3K/mo — alerts could exceed | Daily throttle (`last_fired_at < now() - 24h`) caps at 1 email/alert/day. 100 users × 2 alerts × 30 days = 6K; stay under by limiting 1 alert per portfolio in v1. |
| Supabase free tier: 500 MB DB, 50K MAU, 2 GB bandwidth | Way beyond v1 needs. Re-evaluate at 500+ WAU. |
| Magic-link deliverability (spam filters) | Supabase uses AWS SES; rarely an issue. Add a "didn't get it?" retry button. |
| Cron fails silently | GitHub Actions emails owner on failure by default. Add a self-check: "no alerts processed in 48h" → investigate. |
| User deletes alert but `last_fired_at` lingers | `on delete cascade` handles it. |
| Static-export build-time env vars leak into bundle | Only `NEXT_PUBLIC_*` injected; service keys stay in cron secrets. |
